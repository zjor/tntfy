# tntfy â€” Product Requirements (v1)

## Overview

`tntfy` is an open-source, self-hostable HTTP notification service that delivers messages to Telegram. Spiritual successor to `mqtt2telegram`, inspired by `ntfy.sh`. Any application or script can `POST` to the API and have the message arrive in the user's Telegram DM.

```
curl -H "Authorization: Bearer tk_..." \
     -d "Backup successful" \
     https://tntfy.example.com/v1/publish/deploys
```

The Telegram bot is the control plane: users register, manage topics, and rotate tokens entirely via slash commands. There is no web dashboard in v1.

## Goals (v1)

- One-line `curl` to send a notification to your own Telegram DM
- Full topic lifecycle (create, list, rotate token, remove) via bot slash commands
- Text (plain / markdown / HTML), image, and arbitrary file payloads
- Synchronous, predictable delivery â€” caller learns the outcome on the same request
- Persistent record of every published message (text inline; binary as metadata only)
- Single-process deploy: NestJS handles API, bot, and any background work

## Non-goals (deferred)

- Multi-subscriber topics, group/channel binding (single primary chat per user only)
- Message history retrieval API or replay
- Server-side delivery retries or async queue
- Tags, click actions, priorities, attachments-as-attachments
- Web dashboard, landing page, Telegram mini-app
- Helm chart, docker-compose for production, multi-tenant SaaS hosting
- Rate limiting and quotas (rely on Telegram's throttling for v1)

## User journey

1. User opens the bot in Telegram and runs `/start` â€” a user record is created (or reused) and the DM `chat_id` is bound as the primary delivery channel.
2. User runs `/topic-create deploys` â€” bot replies with a fresh bearer token plus copy-paste-ready `curl` and Python snippets pre-filled with the user's topic name and token. Token is wrapped in `<tg-spoiler>`.
3. User pastes the snippet into their app/cron/script. The notification arrives in their Telegram DM.
4. If the token leaks: `/topic-new-token deploys` issues a new one and immediately invalidates the old.
5. If the topic is no longer needed: `/topic-remove deploys` confirms via inline yes/no button, then deletes the topic, its tokens, and its message history.

## API

### `POST /v1/publish/<topic>`

Publishes a message to the topic identified by the path. The bearer token must belong to that exact topic.

**Headers:**

| Header | Required | Notes |
|---|---|---|
| `Authorization: Bearer <token>` | yes | Per-topic token |
| `Content-Type` | yes | Drives Telegram dispatch (see table below) |
| `Filename` | no | Hint for binary uploads; falls back to a generated name |
| `Caption` | no | Optional caption for `image/*` and `application/octet-stream` payloads |

**Content-Type â†’ Telegram method:**

| Content-Type | Telegram method | Notes |
|---|---|---|
| `text/plain` | `sendMessage` | `parse_mode = none` |
| `text/markdown` | `sendMessage` | `parse_mode = MarkdownV2` |
| `text/html` | `sendMessage` | `parse_mode = HTML` |
| `image/*` | `sendPhoto` | jpeg, png, webp, gif |
| `application/octet-stream` and others | `sendDocument` | Generic file |

**Success response (200):**

```json
{
  "id": "m_4f9...",
  "topic": "deploys",
  "telegram_message_id": 12345,
  "delivered_at": "2026-05-03T10:42:11Z"
}
```

**Error responses:**

| Status | `error` | Cause |
|---|---|---|
| 400 | `empty_body` | Text payload with empty body |
| 400 | `format_error` | Telegram rejected markdown/HTML parsing |
| 401 | `missing_token` | No `Authorization` header |
| 401 | `invalid_token` | Token unknown |
| 404 | `topic_not_found` | Token valid but path topic name does not match the token's topic |
| 413 | `payload_too_large` | Exceeds size limit (text 4096 chars, photo 10 MB, document 50 MB) |
| 415 | `unsupported_content_type` | Unrecognized `Content-Type` |
| 502 | `telegram_blocked` | Telegram returned 403 (bot blocked by user) |
| 502 | `telegram_throttled` | Telegram returned 429; response includes `retry_after` |
| 502 | `telegram_failed` | Other Telegram API error; response includes `reason` |

No automatic server-side retry on 502 in v1. The client decides whether to retry.

### `GET /v1/health`

Liveness check. Returns `200 { "status": "ok" }`.

### `GET /docs`

Swagger UI generated by `@nestjs/swagger`. Reflects the live API surface from controller decorators.

## Bot commands

All commands operate on the user identified by their Telegram `user_id`. Commands run in the bot DM; behavior in groups is undefined for v1.

### `/start`

Idempotent. Creates the user record if missing (storing `ext_id`, `username`, `first_name`, `last_name`). On every run, replies with a welcome message that lists all available commands.

### `/help`

Lists all commands and brief usage examples.

### `/topic-create <name>`

Validates the name against `^[a-z0-9][a-z0-9-_]{1,63}$`. Rejects duplicates within the user's topics. Creates `topics` row plus an active `topic_tokens` row, then replies with:

- The topic name
- The bearer token wrapped in `<tg-spoiler>` to discourage shoulder-surfing
- A `curl` example pre-filled with topic, token, and `PUBLIC_BASE_URL`
- A Python example using `requests`

### `/topic-list`

Lists the user's topics: name and creation date. Tokens are not shown â€” `/topic-new-token` is the path to retrieve a fresh one if the user has lost theirs.

### `/topic-new-token <name>`

Hard-deletes the existing token row, inserts a new one, and replies with the new token. Old token starts returning `401` immediately.

### `/topic-remove <name>`

Replies with an inline keyboard `[Yes, delete]` `[Cancel]`. On confirm: hard-delete `topics` row; FK cascades hard-delete `topic_tokens` and `topic_messages` rows. Replies "removed."

### Bot-side errors

- Invalid topic name â†’ reply with the validation rule and an example
- Topic-name conflict on create â†’ reply "you already have a topic 'X'"
- Topic not found on rotate/remove â†’ reply "no topic 'X', see /topic-list"

## Data model

All primary keys are `nanoid` (21 chars, default alphabet, generated in application code). All timestamps are `timestamptz`.

### `users`

| column | type | notes |
|---|---|---|
| `id` | text | nanoid PK |
| `ext_id` | bigint | Telegram `user_id` (== DM `chat_id`); unique |
| `username` | text | nullable |
| `first_name` | text | nullable |
| `last_name` | text | nullable |
| `created_at` | timestamptz | |
| `deleted_at` | timestamptz | nullable; reserved for a future account-deletion flow, unused in v1 |

### `topics`

| column | type | notes |
|---|---|---|
| `id` | text | nanoid PK |
| `user_id` | text | FK â†’ `users.id`, `ON DELETE CASCADE` |
| `name` | text | `UNIQUE(user_id, name)` |
| `created_at` | timestamptz | |

Hard-deleted on `/topic-remove`.

### `topic_tokens`

| column | type | notes |
|---|---|---|
| `id` | text | nanoid PK |
| `topic_id` | text | FK â†’ `topics.id`, `ON DELETE CASCADE` |
| `token` | text | unique; raw value, format `tk_<24 url-safe chars>` |
| `created_at` | timestamptz | |

Hard-deleted on rotation and on topic removal.

### `topic_messages`

| column | type | notes |
|---|---|---|
| `id` | text | nanoid PK |
| `topic_id` | text | FK â†’ `topics.id`, `ON DELETE CASCADE` |
| `kind` | text | `text` / `image` / `file` |
| `format` | text | nullable; `text` only: `text` / `markdown` / `html` |
| `text_body` | text | nullable; for `kind = text` |
| `mime_type` | text | nullable; for `image` / `file` |
| `content_length` | bigint | nullable; for `image` / `file` |
| `filename` | text | nullable |
| `caption` | text | nullable |
| `status` | text | `delivered` / `failed` |
| `telegram_message_id` | bigint | nullable; set when `delivered` |
| `error` | text | nullable; set when `failed` |
| `created_at` | timestamptz | |

Binary payloads are not stored â€” only metadata. Failed deliveries are persisted with `status = failed` and an `error` string.

### Indexes

- `users (ext_id)` unique
- `topics (user_id, name)` unique
- `topic_tokens (token)` unique
- `topic_tokens (topic_id)`
- `topic_messages (topic_id, created_at DESC)`

### Token verification

```
SELECT
  tk.id     AS token_id,
  tp.id     AS topic_id,
  tp.name   AS topic_name,
  u.id      AS user_id,
  u.ext_id  AS chat_id
FROM topic_tokens tk
JOIN topics tp ON tp.id = tk.topic_id
JOIN users  u  ON u.id  = tp.user_id
WHERE tk.token = $1;
```

Application then verifies path topic name matches `tp.name`; mismatch â†’ `404`. The `chat_id` from this lookup is the destination passed to grammY.

## Operational behavior

### Audit logging

Every state-changing operation emits a single structured (JSON) log line. Common shape: `{ ts, level, request_id, op, user_id, ... }`. Bodies and tokens are never logged. Binary payloads log only size and MIME.

| Operation | `op` | Additional fields |
|---|---|---|
| Bot `/start` | `user.create_or_get` | `ext_id` |
| Bot `/topic-create` | `topic.create` | `topic_id`, `name` |
| Bot `/topic-new-token` | `token.rotate` | `topic_id` |
| Bot `/topic-remove` | `topic.delete` | `topic_id`, `name`, `cascaded_messages_count` |
| API publish | `message.publish` | `topic_id`, `message_id`, `kind`, `status`, `telegram_message_id`, `bytes`, `latency_ms` |

Failures share the same shape with an added `error` field.

### Delivery semantics

- Single synchronous attempt per `POST`.
- HTTP returns the Telegram outcome; the persisted `topic_messages` row reflects the same outcome (`delivered` with `telegram_message_id`, or `failed` with `error`).
- No server-side retries.

## Tech stack

- **Runtime:** Node.js (LTS), TypeScript
- **Framework:** NestJS (single process: HTTP API + Telegram bot + any in-process background work)
- **Telegram:** [grammY](https://grammy.dev) via [`@grammyjs/nestjs`](https://github.com/grammyjs/nestjs)
  - Decision: grammY is the only modern, actively-maintained Telegram framework for Node.js as of 2026-05. Telegraf has been unmaintained since 2025-01; `node-telegram-bot-api` is a thin event-emitter without middleware composition.
- **Database:** Postgres 16+
- **ORM/migrations:** Drizzle (TS-first, type-safe SQL, lightweight, works cleanly with NestJS as a plain provider). Switchable during scaffolding if a different choice is preferred.
- **Bot transport:** long-polling for v1 (no public webhook, no domain prerequisite). Webhook mode behind a config flag â€” deferred.
- **API docs:** `@nestjs/swagger` at `/docs`
- **License:** MIT

## Configuration

The NestJS app reads from environment variables:

| Variable | Required | Notes |
|---|---|---|
| `DATABASE_URL` | yes | Postgres connection string |
| `TELEGRAM_BOT_TOKEN` | yes | From BotFather |
| `PUBLIC_BASE_URL` | yes | Used in `/topic-create` snippets, e.g. `https://tntfy.example.com` |
| `PORT` | no | Defaults to 3000 |
